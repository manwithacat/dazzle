# Fitness v1.0.3 — Contract Anchor + Multi-Persona Fan-Out Design Spec

**Status:** approved; ready for implementation plan
**Date:** 2026-04-14
**Predecessors:** v1.0.1 `_build_engine` wiring (released as v0.54.1), v1.0.2 contract-driven Pass 1 walker (released as v0.54.2)

## Goal

Transform v1.0.2's contract walker from "plumbing that demonstrably works" into "plumbing that actually observes the right component under every relevant persona." v1.0.2 shipped the `walk_contract` mission and the strategy-level `component_contract_path` kwarg, but the walker observes whatever page is loaded at Pass 1 time — fresh Playwright bundles show `about:blank`, so contract gates are "evaluated" against empty HTML. v1.0.3 closes two gaps:

- **A — Anchor navigation.** The contract walker needs to know *where* to look. Add an `anchor:` field to component contracts and navigate the Playwright page to that URL before running the walker.
- **B — Multi-persona fan-out.** `/ux-cycle`'s runbook already says "aggregate across personas," but v1.0.2 runs a single anonymous cycle. Add a `personas` kwarg to `run_fitness_strategy` that loops the walker once per persona, logging in as each via QA mode's magic-link flow (#768).

## Non-goals

The following were considered and explicitly deferred to later releases:

- **Induced-regression replay harness.** The `test_support_tickets_induced_regression_is_caught` e2e test stays skipped; v1 success criterion #5 is a v1.0.4 concern.
- **`walk_story` → `walk_plan` unification.** Optional internal refactor from v1.0.2's briefing. Architectural tidiness with no user-visible benefit. Deferred to v1.0.4+.
- **AppSpec-scoped persona auto-derivation.** The "correct" long-term answer for which personas should run against a component — derive from surface scope rules. Requires a component↔surface link that doesn't exist yet. Deferred; v1.0.3 uses caller-provided explicit persona lists instead.
- **AppSpec-scoped URL derivation.** Same blocker — long-term, the URL should come from the surface route the component lives on. v1.0.3 uses an explicit `anchor:` field instead.
- **Persona filtering by access rights.** v1.0.3 runs every persona the caller passes, even if some can't see the component (they'll record access-denied findings). v1.0.4+ can skip inaccessible personas upfront.

## Architecture

```
/ux-cycle Phase B
       │
       ▼
run_fitness_strategy(example_app, project_root,
                     component_contract_path=None,
                     personas: list[str] | None = None)
       │
       │ 1. launch example app subprocess (QA mode enabled)
       │ 2. _setup_playwright → shared browser (_PlaywrightBundle)
       │ 3. personas_to_run = personas or [None]
       │ 4. for persona_id in personas_to_run:
       │      a. context = browser.new_context()   (fresh cookies)
       │      b. page = context.new_page()
       │      c. if persona_id is not None:
       │           _login_as_persona(page, persona_id, api_url)
       │      d. if contract.anchor is not None:
       │           await page.goto(site_url + contract.anchor)
       │      e. per-persona engine = FitnessEngine(
       │             contract_paths=[component_contract_path],
       │             contract_observer=_ContractObserver(page=page),
       │             ...)
       │      f. try:
       │             result = await engine.run()
       │             outcomes.append((persona_id, result))
       │         except Exception as e:
       │             outcomes.append((persona_id, _blocked_outcome(e)))
       │         finally:
       │             await context.close()
       │ 5. final = _aggregate_outcomes(outcomes)
       │ 6. teardown subprocess
       ▼
StrategyOutcome(
    strategy="FITNESS",
    summary="fitness run [admin:r1, editor:r2, ...]: N findings total (per-persona breakdown)",
    degraded=any(degraded),
    findings_count=sum(findings_count),
)
```

### Three new pieces

1. **`ComponentContract.anchor: str | None`** — new optional dataclass field in `src/dazzle/agent/missions/_shared.py`. Populated by `parse_component_contract` when the contract markdown contains a `## Anchor` section. Defaults to `None`, preserving v1.0.2 backwards compatibility.

2. **`_login_as_persona(page, persona_id, api_url)`** — new async helper in `src/dazzle/cli/runtime_impl/ux_cycle_impl/fitness_strategy.py`. Implements the full magic-link login flow:
   - `POST {api_url}/qa/magic-link` with body `{"persona_id": persona_id}` via `page.request.post(...)`
   - Extract `token` from response JSON
   - `await page.goto(f"{api_url}/auth/magic/{token}?next=/")` — server sets session cookie, redirects to `/`
   - Detect failure modes: 404 on generator (QA mode not enabled), 401 on consumer (token rejected), unexpected redirect to `/login` (login failed silently).

3. **Multi-persona loop + anchor navigation in `run_fitness_strategy`** — the existing strategy is extended so it:
   - Spins up the Playwright *browser* once per strategy run (shared)
   - Creates a fresh *context* per persona (isolated)
   - Parses the contract once at strategy entry to know the anchor
   - Navigates to `site_url + anchor` before building the per-persona engine
   - Builds a separate `FitnessEngine` per persona (each engine is one-shot — no refactor needed), each with its own freshly-constructed `_ContractObserver(page=per_persona_page)` so the observer's `page` reference stays valid for the lifetime of that persona's engine and is never reused across personas
   - Aggregates all per-persona `FitnessRunResult`s into one `StrategyOutcome`

## Component boundaries

| Unit | File | Responsibility |
|---|---|---|
| `ComponentContract.anchor` | `src/dazzle/agent/missions/_shared.py` (modified) | New optional `anchor: str \| None = None` dataclass field. |
| `parse_component_contract` extension | same file | Detect `## Anchor` section; extract first non-empty line as URL; fall back to `None` when missing or empty. |
| `_login_as_persona()` | `src/dazzle/cli/runtime_impl/ux_cycle_impl/fitness_strategy.py` (new function) | Two-step magic-link login via Playwright `page.request.post` + `page.goto`. Raises on any failure mode with a distinguishing error message. |
| Multi-persona loop | same file (modified `run_fitness_strategy`) | Iterates personas, creates fresh `browser.new_context()` per persona, builds per-persona engine, runs, records outcome (success OR blocked), closes context. |
| `_aggregate_outcomes()` | same file (new function) | Reduces `list[tuple[str \| None, FitnessRunResult]]` → single `StrategyOutcome`. Sums `findings_count`, OR-reduces `degraded`, formats a per-persona summary string. |
| Anchor navigation hook | same file (modified `_build_engine` path) | After Playwright bundle setup, if contract has an `anchor`, `await page.goto(site_url + anchor)` before constructing `_ContractObserver`. |
| Fresh context per persona | same file (new helper `_build_persona_bundle`) | Takes the shared browser + persona_id, returns a `_PlaywrightBundle`-like struct with fresh context + page. Per-persona teardown in the loop's `finally`. |

## Data flow

### Happy path — single anonymous run (v1.0.2 backwards-compat)

```
caller: run_fitness_strategy(..., component_contract_path=p, personas=None)
  └─ launch subprocess (DAZZLE_QA_MODE=1)
  └─ _setup_playwright(base_url) → bundle (shared browser)
  └─ contract = parse_component_contract(p)
  └─ personas_to_run = [None]
  └─ for persona_id in [None]:
       ├─ context = bundle.browser.new_context()
       ├─ page = await context.new_page()
       ├─ # persona_id is None — skip login
       ├─ if contract.anchor: await page.goto(site_url + contract.anchor)
       ├─ per_engine = FitnessEngine(contract_paths=[p],
       │                             contract_observer=_ContractObserver(page))
       ├─ result = await per_engine.run()
       ├─ outcomes.append((None, result))
       └─ await context.close()
  └─ final = _aggregate_outcomes(outcomes)
  └─ await bundle.close()
  └─ _stop_example_app(handle)
  └─ return final
```

### Happy path — multi-persona

Same flow, but `personas_to_run = ["admin", "editor", "viewer"]`. Loop runs three times. Each iteration has its own context, its own login, its own engine. All three engines share the same subprocess and the same Playwright browser, and each iteration's context is torn down before the next begins.

### Error handling

| Failure mode | Where it fires | Teardown | Loop behavior |
|---|---|---|---|
| Subprocess fails to start | `_launch_example_app` raises | nothing to tear down | Strategy raises — no loop entered |
| Playwright setup fails | `_setup_playwright` raises | outer `finally` → `_stop_example_app` | Strategy raises — no loop entered |
| Contract file missing | `parse_component_contract` raises `FileNotFoundError` at strategy entry | outer `finally` | Strategy raises before loop |
| Contract has no `anchor` | `contract.anchor is None` — expected; walker observes whatever page is loaded | N/A | Normal — v1.0.2 behavior preserved |
| `_login_as_persona` fails | `POST /qa/magic-link` 404 (QA mode off), or token rejected, or session redirect loops back to `/login` | inner `finally` closes context | Current persona recorded as BLOCKED; loop continues |
| Anchor navigation fails | `page.goto` raises timeout/network error | inner `finally` | Current persona BLOCKED; loop continues |
| `engine.run()` raises | `_EngineProxy.run()` closes its sub-bundle; inner `finally` closes the persona context | both close | Current persona BLOCKED; loop continues |
| All personas fail | outcomes list populated with all-BLOCKED | All contexts closed, subprocess torn down | Strategy returns an aggregated "all BLOCKED" `StrategyOutcome` (does not raise) |

**Fail-fast vs continue:** v1.0.3 uses **continue mode** — one persona's failure is recorded as a BLOCKED outcome but does not abort the remaining personas. This gives Phase B the most information per invocation. Callers who need fail-fast can inspect `StrategyOutcome.findings_count` or a future per-persona breakdown field.

### Aggregation format

Single-persona summary (unchanged from v1.0.2):
```
fitness run r1: 0 findings, independence=0.000
```

Multi-persona summary (new, matches when `len(outcomes) > 1`):
```
fitness run [admin:r1, editor:r2, viewer:r3]: 4 findings total (admin=0, editor=4, viewer=0), independence=0.850 (max)
```

The `StrategyOutcome` fields (`strategy`, `summary`, `degraded`, `findings_count`) stay identical in shape. Callers that only care about the aggregate numbers work unchanged; callers that want the per-persona breakdown parse it from the summary string for v1.0.3 (a structured `per_persona` dict is a v1.0.4 add).

## Contract anchor syntax

Extend the ux-architect contract markdown with one new optional section. The parser already reads `## Quality Gates`, `## Anatomy`, `## Primitives invoked` — `## Anchor` fits the same pattern.

**Example:**
```markdown
# auth-page

## Anchor

/login

## Quality Gates

1. Card is centered in viewport
2. Submit button uses primary color
3. Error div starts hidden

## Anatomy
...
```

**Parser behavior:**
- If `## Anchor` section present with non-empty body: `contract.anchor = <first non-blank line stripped>`
- If section absent OR body empty: `contract.anchor = None`
- If section present with multiple non-blank lines: parser takes the first non-blank line and ignores the rest (no diagnostic — the rest is treated as comment text)
- Leading/trailing whitespace trimmed; empty `## Anchor\n\n\n## Quality Gates` is treated as absent

**35+ existing contracts lack an anchor.** They are not modified in v1.0.3 source code. A separate one-shot data migration (outside the v1.0.3 code scope) populates the `## Anchor` section on each contract file. This lets v1.0.3 ship the *capability* without coupling its release to the *data migration*. Contracts without an anchor fall back to v1.0.2 behavior (observe whatever page is loaded), preserving backwards compatibility.

2. **Task 2 — Anchor navigation hook in `_build_engine`** (~20 LOC src, ~40 LOC test). Uses Task 1's field. Thin.

Tasks 1-3 can be reviewed independently. Task 4 is where the complexity concentrates and gets the closest review attention.

## Open questions / known limitations

- **v1.0.3 does NOT auto-derive which personas should run for a given component.** The caller must pass an explicit list. A component that legally only allows "admin" will record failures for any other persona the caller includes. This is the v1.0.4 "AppSpec-scoped persona auto-derivation" work.
- **v1.0.3 does NOT validate the anchor URL against a routing table.** An anchor that points at a 404 page gets navigated to, the walker observes the 404 page, and gates mostly fail with "error: <missing element>" observations. This is the expected behavior: a contract whose anchor is wrong *should* produce obvious walker failures. A validator that pre-checks anchors against `dazzle validate` output is a v1.0.4+ concern.
- **The contract migration (populating `## Anchor` on existing contracts) is out of scope for the v1.0.3 source code.** It is expected as a separate one-shot data commit.
- **Per-persona breakdown in `StrategyOutcome` is a string format, not a structured dict.** Callers that need programmatic per-persona access parse the summary string. v1.0.4 can add a `per_persona: dict[str, FitnessRunResult]` field.
- **Playwright browser is shared across personas.** This is an intentional performance choice (browser launch is slow). Context isolation is sufficient for cookie/session separation. If a future persona needs a different browser engine (e.g., Firefox vs Chromium for cross-browser gates), that requires a larger refactor.
