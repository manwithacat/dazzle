# Feedback Loop System

> **Issue:** #685 (extends CRUD integration)
> **Status:** Approved
> **Date:** 2026-03-25
> **Depends on:** Feedback CRUD integration (synthetic surfaces, done)

## Goal

Close the human→agent feedback loop. Users submit feedback via the widget; agents read, triage, fix, and resolve it via MCP tools. A CLI provides human override. The knowledge graph and bootstrap encourage adoption on every new auth-enabled Dazzle app.

## Design Decisions

- **Simplified workflow**: `triage` + `resolve` as the two primary actions (maps to `new→triaged` and `→resolved` in the 7-state machine). Full lifecycle states exist but are not surfaced as top-level operations.
- **Agent-primary, human-override**: MCP tool is the main interface; CLI is a thin wrapper calling the same impl.
- **Consistent architecture**: EDIT surface generated by the linker (same synthetic pattern as CREATE and LIST). No bypassing the surface/route pipeline.

## Changes

### 1. Linker — Third Synthetic Surface (`feedback_edit`)

Add `_build_feedback_edit_surface()` in `src/dazzle/core/linker.py`, called alongside the existing CREATE and LIST surface builders.

- `name: "feedback_edit"`, `mode: edit`, `uses entity FeedbackReport`
- Sections with editable fields: `status`, `assigned_to`, `agent_notes`, `agent_classification`, `related_entity`, `related_story`
- Access: `require_auth=True`, `allow_personas=["admin", "super_admin"]`
- Generates `PUT /feedbackreports/{id}` via the normal surface converter

Complete surface table after this change:

| Surface | Mode | Route | Access |
|---------|------|-------|--------|
| `feedback_create` | CREATE | `POST /feedbackreports` | any authenticated |
| `feedback_admin` | LIST | `GET /feedbackreports` | admin, super_admin |
| `feedback_edit` | EDIT | `PUT /feedbackreports/{id}` | admin, super_admin |
| *(auto-generated)* | DELETE | `DELETE /feedbackreports/{id}` | *(from LIST surface)* |
| *(auto-generated)* | READ | `GET /feedbackreports/{id}` | *(from LIST surface)* |

### 2. MCP Tool — `feedback` handler

New handler at `src/dazzle/mcp/server/handlers/feedback.py`, registered in the consolidated tools.

Four operations:

| Operation | Parameters | Behaviour |
|-----------|-----------|-----------|
| `list` | `status?`, `category?`, `severity?`, `limit?` (default 20) | GET /feedbackreports with query filters. Returns paginated list: id, category, severity, description, status, page_url, reported_by, created_at. |
| `get` | `id` (required) | GET /feedbackreports/{id}. Returns full report with all fields. |
| `triage` | `id` (required), `agent_notes?`, `agent_classification?`, `assigned_to?` | PUT /feedbackreports/{id} with `status: "triaged"` plus optional triage fields. State machine validates `new→triaged`. |
| `resolve` | `id` (required), `agent_notes?`, `resolved_by?` | PUT /feedbackreports/{id} with `status: "resolved"` plus optional resolution fields. State machine validates `triaged→resolved` or `in_progress→resolved`. |

The handler reads the server URL from the runtime file (`.dazzle/runtime.json`) and calls the CRUD endpoints over HTTP — same pattern as other MCP handlers that interact with the running server. Auth, CSRF exemption, state machine validation, and access control all apply through the normal pipeline.

### 3. CLI — `dazzle feedback` command group

New file `src/dazzle/cli/feedback.py` with `feedback_app = typer.Typer(...)`, registered in `src/dazzle/cli/__init__.py`.

| Command | Flags | Output |
|---------|-------|--------|
| `dazzle feedback list` | `--status`, `--category`, `--severity`, `--json` | Table or JSON of reports |
| `dazzle feedback get <id>` | `--json` | Full report detail |
| `dazzle feedback triage <id>` | `--notes`, `--assign`, `--classify`, `--json` | Confirms new→triaged |
| `dazzle feedback resolve <id>` | `--notes`, `--resolved-by`, `--json` | Confirms →resolved |
| `dazzle feedback delete <id>` | `--json` | Confirms deletion |
| `dazzle feedback stats` | `--json` | Counts by status, category, severity |

Impl functions live in a shared module (`src/dazzle/cli/feedback_impl.py`) imported by both CLI and MCP handler. They call the CRUD endpoints on the running server via HTTP.

### 4. Knowledge Graph — Semantics KB

Add feedback concepts to `src/dazzle/mcp/semantics_kb/` (new file `feedback.toml` or entries in `ux.toml`):

**Concepts:**
- `feedback_widget` — What it is, how to enable, what it auto-generates (entity + 3 surfaces + routes)
- `feedback_report` — The auto-generated entity, its fields, state machine, access rules
- `feedback_loop` — The human→widget→agent→resolve workflow pattern

**Workflow:**
- `feedback_triage_workflow` — Steps: list open feedback → read detail → fix issue → resolve with notes

**Example:**
- DSL snippet showing `feedback_widget: enabled` with sub-keys
- MCP tool usage showing list → triage → resolve sequence

If using a new `feedback.toml` file, add `"feedback.toml"` to the `TOML_FILES` list in `src/dazzle/mcp/semantics_kb/__init__.py` (files not in this list are never seeded). Bump `SEED_SCHEMA_VERSION` in `src/dazzle/mcp/knowledge_graph/seed.py` to force re-seed.

### 5. Bootstrap — Recommend Feedback Widget

In `src/dazzle/mcp/server/handlers/bootstrap.py`, add feedback_widget to the generation workflow:

- Add a step (~step 9) in the agent instructions: "If auth is enabled, add `feedback_widget: enabled` after the app declaration to enable the human→agent feedback loop."
- Conditional on auth: the widget requires authentication to attribute feedback to users. If `auth: enabled` is not present, skip the recommendation.

### 6. CSRF — Extend Exemption to Sub-paths

The exact path `/feedbackreports` is already in `exempt_paths` (covers `POST /feedbackreports`). Add `/feedbackreports/` to `exempt_path_prefixes` in `src/dazzle_back/runtime/csrf.py` so that `PUT /feedbackreports/{id}` and `DELETE /feedbackreports/{id}` are also exempt. The MCP tool and CLI call these endpoints over HTTP without browser CSRF tokens (they authenticate via session cookies obtained from the `/__test__/` or `/auth/login` endpoints).

### 7. State Machine — Add `triaged→resolved` Shortcut

Add a `triaged → resolved` transition to `_build_feedback_report_entity()` in the linker. Most feedback is quick-fix — forcing `triaged→in_progress→resolved` adds friction for the agent workflow. The `in_progress` state remains available for longer investigations but is not required.
