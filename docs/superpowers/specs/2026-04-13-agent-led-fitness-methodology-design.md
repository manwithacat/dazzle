# Agent-Led Fitness Methodology — Design

**Status:** Draft for review
**Date:** 2026-04-13
**Owner:** Dazzle team (self-hosted) + Dazzle users (via MCP)
**Supersedes:** N/A (new subsystem)
**Depends on:** DSL + /bootstrap + template compiler + DazzleAgent framework + mcp__dazzle__db

## 1. Overview

The **Agent-Led Fitness Methodology** is a continuous V&V loop that evaluates whether a Dazzle app is *fit for the purpose described by its founder's spec*. Unlike mechanical QA (does the form submit? does the route render?) it answers semantic questions: **does this perform useful work for the persona? does it change entropy in ways congruent with the spec? would a first-time user figure it out without training?**

The methodology triangulates three corners — `spec.md` (the founder's natural-language oracle), the `DSL` (the intermediate representation), and the `running app` — and emits a fitness signal that feeds back into the codebase either autonomously (hard correction) or via human-gated PR (soft correction).

It serves two use cases with a single shared engine:

- **Dazzle team (dogfooding):** integrated into `/ux-cycle` as a new strategy. Atomic per example app. Drives example quality + surfaces framework gaps.
- **Dazzle users (their own projects):** exposed via `mcp__dazzle__fitness` MCP tool. Same engine, scoped to their current project. Same methodology, thin invocation shim.

## 2. Goals

1. Distinguish "the spec was wrong", "the DSL was wrong", and "the implementation was wrong" — localise the disagreement, not just detect it.
2. Catch silent data-loss failures that pure-DOM observation would miss (the "motion without work" problem).
3. Surface implicit capabilities the spec mentions but no DSL story covers.
4. Produce findings in a format that supports autonomous correction (hard mode) or human review (soft mode).
5. Do all of this without asking the founder to write specs, stories, or process definitions in framework vocabulary they don't have.
6. Scale from MVP (one spec, a handful of personas, a few entities) to stable (dozens of personas, hundreds of stories, years of drift).

## 3. Non-goals

1. **Not a test framework.** It's a usability methodology that happens to produce findings. It does not replace unit tests, integration tests, or the existing `dazzle ux verify --contracts` mechanical checks.
2. **Not a security auditor.** RBAC gaps, injection attacks, CSRF edges are out of scope. Fitness checks functional correctness, not threat model correctness.
3. **Not a performance profiler.** Findings about latency, query plans, or N+1 queries are out of scope.
4. **Not a replacement for human product review.** Soft correction mode routes findings to a human PR queue precisely because some judgments are not delegable.

## 4. Named design principles

### 4.1 Deterministic Centre, Agentic Edges

> The DSL is the deterministic source of truth for the app. All difficult agentic cognition happens BEFORE the DSL (analysing a messy spec) and AFTER the DSL (exploring a built app in a near-human behavioural mode). The DSL → running app pipeline stays mechanical.

Consequence: Pass 1 (story walk) uses scripted Playwright runs, not LLM-as-agent. LLM-as-agent cognition is reserved for Pass 2 (spec cross-check + free-roam exploration) where judgment is actually required. This reduces token burn and removes non-determinism from the parts of the pipeline that should be reliable.

### 4.2 Agent-Ergonomic Tooling

> Tools that are too cognitively expensive for human developers can be the right tools for agents. Invest in agent-first infrastructure that leverages this asymmetry — WAL inspection, trigger-based change logs, structured protocol traces, content-addressable change streams. Agents pay no memory cost for complexity; they pay a verification cost for ambiguity. Trade one for the other wherever possible.

Consequence: the `FitnessLedger` target architecture (v1.2) treats the PostgreSQL logical decoding stream (WAL) as the authoritative observation substrate. Humans rarely use WAL directly because of parsing overhead; agents don't care. This is the strong form of the "compact DB-diff" requirement — not a polling approximation but a real change feed.

### 4.3 Ask for Recognition, Not Generation

> Founders can recognize when their intent is misrepresented, even when they can't write the correct representation themselves. Design founder-facing loops around what they CAN do (recognize, correct paraphrases, react) rather than what they can't (write specs, author stories, disambiguate upfront). The agent generates; the founder recognizes.

Consequence: DSL stories are generated by /bootstrap (founder never writes them). An adversarial re-reader audits them every cycle. When feedback is needed, an agent paraphrases the story back to the founder in plain English — the founder either confirms or corrects the paraphrase. No framework vocabulary required at any step.

## 6. The three passes

### 6.1 Pass 1 — Story Walker (deterministic verification)

**Inputs:** DSL stories, running app, persona credentials, expected-state manifest (derived from each story's postconditions).

**Behaviour:** For each story belonging to persona P:
1. Authenticate as P
2. Run the scripted Playwright sequence derived from the story's action steps
3. Record observed DOM + HTTP responses
4. At mission end, diff ledger against story-derived expectation
5. Emit Pass-1 findings for any mismatch

**What it catches:** implementation bugs, silent discards (truth-diff), progress-failures.

**What it does NOT catch:** capabilities the founder wanted but the DSL never declared. (That's Pass 2a's job.)

**Cognition profile:** zero LLM calls. Deterministic script execution. Cheap and reliable. Runs every commit at minimum, every cycle definitely.

### 6.2 Pass 2a — Spec Cross-check (agentic validation)

**Inputs:** `spec.md`, DSL story list.

**Three sub-steps, run serially:**

**Sub-step A — spec_extractor:**
- LLM reads `spec.md` with prompt "extract the list of jobs-to-be-done implied by this spec; for each job, note the persona who performs it"
- Output: structured capability list

**Sub-step B — cross_check:**
- For each extracted capability, search the DSL story list for a story that covers it
- Emit findings for capabilities with no matching story (spec→no-story coverage gap)
- Also emit findings for stories with no matching capability (over-implementation; possible agent hallucination at /bootstrap time)

**Sub-step C — adversary (the independence layer):**
- LLM reads `spec.md` AGAIN with a different framing: "find any capability implied by this spec that's NOT in the list [<Sub-step A's output>]"
- Different prompt = different noise profile = catches things the first extraction missed
- Output: additional findings surfaced by the second reading

**Cognition profile:** 2-3 LLM calls per cycle (spec extraction + adversarial re-read + optional per-story alignment). No Playwright needed.

### 6.3 Pass 2b — Human Proxy (agentic verification, behavioural)

**Inputs:** running app, persona assignment, high-level intent derived from spec.

**Behaviour:** The agent is given a persona (sarah@support.example.test), a high-level intent ("your job today is to get work done as yourself"), and a step budget. It drives the app via Playwright using the **EXPECT/ACTION/OBSERVE protocol**:

For every intended action, the agent emits three fields in sequence:

```
EXPECT: "when I click Save, I expect a confirmation toast and the ticket
         to appear in my assigned list"
ACTION: [tool call — click button]
OBSERVE: {status: surprise|match|error,
          text: "the toast appeared but the list didn't update"}
```

The mission ends when the agent reports "I've accomplished what I set out to do" or when the step budget is exhausted.

**Why EXPECT matters:** the fitness signal is `OBSERVE - EXPECT`, not `OBSERVE` alone. A toast that appears without the list updating is ONLY a finding if someone expected the list to update. The expectation provides the reference signal against which observations are scored.

**What it catches:** lifecycle violations (motion without progress), UX surprises, dead-end navigation, onboarding friction.

**Cognition profile:** full-budget agent mission. Most expensive step. 30-50 steps typical.

## 7. FitnessLedger contract

### 7.1 Abstraction surface

```python
class FitnessLedger:
    def open(self, run_id: str) -> None: ...
    def record_intent(self, step: int, expect: str, action_desc: str) -> None: ...
    def summarize(self) -> FitnessDiff: ...
    def close(self, rollback: bool = False) -> None: ...
```

Agents see only these four methods. Implementation underneath is a config decision, not an agent-prompt decision.

### 7.2 `FitnessDiff` schema

```python
@dataclass
class FitnessDiff:
    run_id: str
    steps: list[LedgerStep]
    created: list[RowChange]
    updated: list[RowChange]
    deleted: list[RowChange]
    progress: list[ProgressRecord]  # entity lifecycle advance/stagnation
    semantic_repr_config: dict       # which fields projected per table

@dataclass
class LedgerStep:
    step_no: int
    expected: str                    # EXPECT field from agent
    action_summary: str              # ACTION summary
    observed_ui: str                 # OBSERVE field from agent
    observed_changes: list[RowChange]  # what actually changed in the DB
    delta: Delta                     # EXPECTED - OBSERVED (the fitness signal)

@dataclass
class RowChange:
    table: str
    row_id: str
    kind: Literal["insert", "update", "delete"]
    semantic_repr: str               # human-readable compact projection
    field_deltas: dict[str, tuple]   # {field: (before, after)}

@dataclass
class ProgressRecord:
    entity: str
    row_id: str
    states_visited: list[str]
    states_declared: list[str]
    verdict: Literal["advanced", "stagnated", "regressed"]
```

The `semantic_repr` is a human-readable compact projection of a row (not the full row). Which fields are projected is DSL-driven at build time via a `fitness.repr_fields` declaration per entity. Defaults to DSL's `list_columns` if not declared.

### 7.3 Implementation roadmap

| Version | Strategy | DB isolation | Cost | Fidelity |
|---|---|---|---|---|
| **v1** | Snapshot diff: `SELECT id, <repr_fields> FROM <interesting tables>` before + after, diff rows | None (shares connection pool) | O(N) per snapshot where N = row count | ✅ covers INSERT/UPDATE/DELETE ❌ misses transaction grouping ❌ misses async-worker changes if they happen outside the mission window |
| **v1.1** | SAVEPOINT wrapping: test-mode middleware binds the mission to a single connection; SAVEPOINT before, ROLLBACK after | Full transactional isolation | O(changes) | ✅ no state pollution ✅ repeated runs ❌ requires app cooperation (single-connection mode); breaks with background workers |
| **v1.2** (target) | Logical replication slot: WAL stream is the observation substrate | None (works with normal pool) | O(WAL volume) | ✅ exact fidelity ✅ transaction grouping for free ✅ captures background workers ✅ agent-ergonomic (WAL output is structured, not prose) |

v1 ships first. v1.1 gets user apps clean isolation. v1.2 is the target architecture (agent-ergonomic tooling applied).

## 8. Finding schema

Findings are the durable output of every fitness run. Each finding is a row in `fitness-backlog.md` and has this shape:

```
Finding:
  id:                  FIND-NNN (append-only counter)
  created:             ISO timestamp
  run_id:              FK to fitness-run (ephemeral)
  cycle:               FK to /ux-cycle (if dev-team) or N/A (if user)
  axis:                coverage | conformance
  locus:               implementation | story_drift | spec_stale | lifecycle
  severity:            critical | high | medium | low
  persona:             which persona exercised this
  capability_ref:      spec clause excerpt OR story ID
  expected:            EXPECT field (for conformance findings)
  observed:            OBSERVE field (for conformance findings)
  evidence_ledger:     ref into .dazzle/fitness-ledger/<run-id>.jsonl
  disambiguation:      bool (did corrector need to pick between interpretations?)
  status:              PROPOSED | ACCEPTED | IN_PROGRESS | FIXED | VERIFIED | REJECTED
  route:               hard | soft (set by corrector)
  fix_commit:          git SHA when fixed
```

**Axis × locus matrix** for guidance:

| axis \ locus | implementation | story_drift | spec_stale | lifecycle |
|---|---|---|---|---|
| **coverage** | rare (usually "capability not wired up") | common (story missing, capability in spec) | common (spec mentions X, nobody built it) | rare |
| **conformance** | common (form silently drops data) | common (story says X, app does Y) | rare (spec wrong, reality right) | common (motion without progress) |

## 9. Corrector

Two-gate hard/soft routing:

```python
def route_finding(finding: Finding, maturity: str) -> Route:
    # Gate 1: maturity kill-switch
    if maturity == "stable":
        return Route.SOFT

    # Gate 2: per-finding disambiguation check
    if finding.disambiguation:
        return Route.SOFT

    return Route.HARD
```

**Hard mode (autonomous commit):**
- Corrector reads the finding + relevant DSL/template/code
- Generates a fix
- Commits directly with message `fitness: auto-fix FIND-NNN — <short description>`
- Emits signal for regression comparator

**Soft mode (human gate):**
- Corrector generates a fix proposal
- Files it as a PR or as a TODO row in the fitness-backlog
- Does NOT commit
- Human reviews and accepts/rejects

**Disambiguation flag** is set by the corrector itself during fix generation. Plumbing: the corrector's internal prompt ends with `did you need to choose between plausible interpretations? → yes | no`. The answer gates routing.

## 10. Regression detection (non-monotonic fitness)

Corrector commits can make things worse. The engine runs a comparator step on every cycle:

```python
def compare_runs(current: FitnessDiff, previous: FitnessDiff) -> RegressionReport:
    new_findings = current.findings - previous.findings
    fixed_findings = previous.findings - current.findings
    persistent_findings = current.findings & previous.findings

    if new_findings and previous_cycle_had_hard_correction:
        escalate_to_pr_or_revert()

    return RegressionReport(new=new_findings, fixed=fixed_findings,
                            persistent=persistent_findings)
```

**Regression policy:**
- If the previous cycle applied a hard correction AND the current cycle has new findings that didn't exist before → the correction introduced a regression. Escalate: either automatic revert OR convert the original finding to a PR for human review.
- If the same finding persists for N consecutive cycles → mark `BLOCKED` (the corrector can't fix it alone).
- If findings oscillate (appear, disappear, reappear) → mark `REGRESSION` class on the affected finding (indicates instability in the corrector's fix strategy).

This closes the control loop and prevents the pipeline from being a "plausible-looking improvement loop that's actually a random walk".

## 11. Paraphrase-confirm loop

Triggered on-demand (not per-cycle). Flow:

1. Agent selects a batch of stories (e.g., all stories containing "support agent" verbs, or all stories with `status=provisional`)
2. For each story, LLM generates a plain-English paraphrase with **no framework vocabulary**:
   - ❌ "When a support-agent persona invokes the transition action on an entity in state=open..."
   - ✅ "When Sarah from support marks a ticket as resolved..."
3. Paraphrases are presented to the founder (via MCP tool response, or in /review-stories for dazzle-team)
4. Founder responds per paraphrase: `confirm | correct: <text> | unclear`
5. Confirmations mark the story as `audited=true`; corrections become `story_drift` findings that route to /bootstrap re-runs.
6. The adversary (Pass 2a sub-step C) tightens its audit around audited stories (tolerates less drift) and relaxes around provisional ones.

**Graduation:** stories that survive N paraphrase-confirm rounds across time without correction are promoted from `provisional` to `confirmed`. Confirmed stories become higher-trust references for verification.

### 12.3 External aggregation (Dazzle team only)

A separate aggregator job, run weekly:

- Reads `fitness-backlog.md` rows committed in the past week across all example apps
- Groups findings thematically (by category, by affected entity, by shared root cause)
- Produces a single GitHub issue per theme with a narrative summary
- Issues get labels: `fitness-report`, `coverage` or `conformance`, `mvp`/`beta`/`stable`

This is marketing/community signal, not internal tracking. The durable internal record stays in `fitness-backlog.md`.

## 15. Open questions / deferred decisions

1. **Persona coordination:** sequential per persona in a single cycle, or parallel? v1 uses sequential to keep the lock pattern simple; v1.2 can parallelize if Playwright isolation is clean enough.

2. **Mission budget:** step count is plausible (~30-50 per Pass 2b mission) but not empirically validated. First cycles will probably adjust based on what budget produces useful findings without exhausting token budget.

3. **Adversary model variance:** how different does the re-reader's prompt need to be from the original extractor to produce genuinely independent signal? Empirically validated in v1.1 — if the adversary's findings correlate too strongly with the original, the independence assumption is broken and we need a different approach (e.g., different model, different temperature, different persona framing).

4. **Semantic repr field selection:** defaults to `list_columns` from the DSL, but `list_columns` is UI-optimized (things users see), whereas semantic repr should be domain-optimized (things that matter for fitness). A dedicated `fitness.repr_fields` DSL declaration may be needed per entity.

5. **Cycle cadence:** how often does FITNESS rotate in alongside other /ux-cycle strategies? Every 3rd cycle? Every 10th? TBD — tune based on signal-to-noise observed in v1.

6. **`--success` / `--warning` design tokens:** called out multiple times in the UX cycle backlog (widget:file, toast, base-layout). Should be added to `design-system.css` — independent of the fitness methodology but referenced by some of its reporting UI.

## 16. Success criteria (v1)

The methodology is considered successful if:

1. It detects a real conformance finding in an existing example app that the ux-cycle wouldn't catch (proof of differentiated value over mechanical QA).
2. It localises at least one finding correctly across the disagreement-locus axes (proof of triangulation value).
3. Its hard-mode corrector produces a committable fix for at least one detected finding without manual editing.
4. Its regression comparator correctly detects at least one induced regression (self-validation via an intentionally buggy correction).
5. It runs to completion in under 10 minutes per example app on the Dazzle team's local hardware (scalability signal).

## 17. Appendix: cognitive framing (Kalman filter metaphor)

The methodology is, in essence, a **Kalman filter over system correctness**. Three noisy sensors measure the same latent variable ("is the system right?"):

- `spec.md` — what the founder intended (noisy because founders can't articulate precisely)
- DSL stories — what /bootstrap thought the founder meant (noisy because LLMs interpret)
- Observed behaviour — what the running app actually does (noisy because implementation bugs, race conditions, etc.)

`FitnessDiff` is the **innovation** — the disagreement between predicted and observed. The corrector is the **update step**, applying fixes that reduce the innovation on the next cycle.

Kalman filters work because the sensors have **independent noise**. If your three corners share a common origin (e.g., /bootstrap writes both the DSL stories and the spec interpretation), the whole thing degrades to a single noisy estimate and you can't localise disagreement. Guarding corner independence is the design discipline — it's why the adversary exists, why the paraphrase loop exists, why we care about who authors the stories.

Over enough cycles, a well-tuned Kalman filter converges toward the truth. A well-tuned fitness loop converges toward a system that is fit for its founder's purpose.
