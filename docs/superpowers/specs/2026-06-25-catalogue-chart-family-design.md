# UX Catalogue — complete the chart family (Sub-project B, slice 1)

**Status:** Approved design — ready for implementation plan.
**Context:** The published UX catalogue (Sub-project A, v0.86.25+) covers 10 of 34 display modes. This slice completes the **chart family** (10 → 17) and folds the fidelity marker into the manifest (a light single-source-of-truth step). Sub-project B's later slices: the list/card modes, the config-driven modes, real aggregates over demo_data.

## Goal

Add the 7 remaining chart-family modes to the catalogue so every chart display is shown — `line_chart`, `area_chart`, `sparkline`, `histogram`, `box_plot`, `radar`, `funnel_chart` — and move the per-mode fidelity **marker into the manifest** so the fidelity test derives from `CATALOGUE_MANIFEST` (no hand-maintained parametrize). No new render code (these modes already have builders); this is **content + a test refactor**.

## Design decisions (locked)

1. **7 chart modes**, each = a DSL region in the `ux_catalogue` fixture workspace + a `CATALOGUE_MANIFEST` entry (sample data + description + marker).
2. **Marker into the manifest** — `CatalogueEntry` gains `"marker": str`; the fidelity test iterates `CATALOGUE_MANIFEST` asserting each entry's marker renders (replacing the hand-listed parametrize). The 3rd edit-place folds into the manifest.
3. **Per-mode probe-then-pin** — the exact canned data shape + the real `dz-*` marker for each mode is verified by a quick render probe (grep the emitted class), not guessed — the same empirical loop used for the first 8 modes.
4. **The fixture DSL region stays** — it is the real DSL the catalogue renders; the manifest carries only the sample data + description + marker.

## Non-goals (deferred to later B slices)

- The list/card modes (timeline, tree, queue, activity_feed, status_list, progress) and the config-driven modes (cohort_strip, day_timeline, task_inbox, entity_card, profile_card, detail, confirm_action_panel, search_box, action_grid, pipeline_steps, diagram, tabbed_list).
- Real aggregate computation over demo_data (the harness keeps using canned buckets / sample items).
- A full registry that also generates the fixture DSL.

## Architecture — content + a test refactor

### Fixture (`fixtures/component_showcase/dsl/app.dsl`)
Add 7 regions to the `ux_catalogue` workspace (Box entity has `name`, `team` enum, `status` enum, `latency_ms` int, `error_rate` decimal, `target_ms` int, `opened_at` datetime):
- `cat_radar` — `display: radar`, `group_by: team`, `aggregate: count`.
- `cat_funnel` — `display: funnel_chart`, `group_by: status`, `aggregate: count`.
- `cat_line` — `display: line_chart`, `group_by: bucket(opened_at, day)`, `aggregate: count`.
- `cat_area` — `display: area_chart`, `group_by` per what the area builder needs (single time-bucket, or `[bucket(opened_at, week), team]` if multi-series is required — pinned by probe).
- `cat_sparkline` — `display: sparkline`, `group_by: bucket(opened_at, day)`, `aggregate: count`.
- `cat_histogram` — `display: histogram`, `value: latency_ms` (the `heatmap_value` field) over items.
- `cat_box_plot` — `display: box_plot`, `group_by: team`, `value: latency_ms` over items.

### Manifest (`src/dazzle/testing/ux_catalogue_manifest.py`)
- Add a `"marker"` key to **every** `CATALOGUE_MANIFEST` entry (the 10 existing + the 7 new) — the `dz-*` class the mode emits.
- Add the 7 new entries with `description`, `marker`, and the right sample shape:
  - `canned_buckets` for radar/funnel/line/area/sparkline (bucket dicts; time modes use time-labelled buckets with a `bucket` iso key).
  - `sample_items` = `_BOXES` for histogram/box_plot (they read a numeric column from items).

### Fidelity test (`tests/unit/test_ux_catalogue.py`)
- Replace the hand-listed `test_mode_renders_primitive` parametrize with a manifest-driven version:

```python
@pytest.mark.parametrize("name", sorted(CATALOGUE_MANIFEST))
def test_mode_renders_primitive(name: str) -> None:
    html = _render(name)
    marker = CATALOGUE_MANIFEST[name]["marker"]
    assert html.strip(), f"{name} rendered empty"
    assert marker in html, f"{name} missing {marker!r}"
    assert "dz-empty" not in html, f"{name} fell through to an empty state"
```

- Keep `test_every_catalogue_region_has_a_manifest_entry` (region set == manifest keys).

### Generated page (`docs/reference/ux-catalogue.md`)
- Regenerated by `scripts/gen_ux_catalogue.py` (unchanged) — the 7 new modes auto-appear in declared order. The CSS bundle already covers chart classes (it's generated from the component CSS).

## Testing (TDD per mode, then the refactor)

1. **Marker-into-manifest refactor** — add `"marker"` to the 10 existing entries; switch the fidelity test to iterate the manifest; `pytest tests/unit/test_ux_catalogue.py` → PASS (10 modes still green, now manifest-driven).
2. **Per new mode (×7)** — probe the render (grep the `dz-*` class + confirm no empty-state), add the DSL region + manifest entry (data + marker + description), `pytest tests/unit/test_ux_catalogue.py -k <name>` → PASS. Group a few modes per task.
3. **Validate + regen + full suite + ship** — `cd fixtures/component_showcase && dazzle validate` exit 0; regen the page (`scripts/gen_ux_catalogue.py`); `pytest tests/ -m "not e2e"` (ignore the 3 `test_fuzzer_oracle` pollution failures); the `test_dazzle_validate_drift` gate stays within the warning grace (7 new regions add warnings); ruff + bare `mypy src/dazzle` clean; `/bump patch` with CHANGELOG (`### Added` the 7 chart modes + the manifest-driven fidelity test); commit + tag + push; watch CI + docs deploy green; confirm the live catalogue shows 17 modes.

## Complexity / risk notes

- **No render-code change** — the 7 modes already have builders (line/area/sparkline via `_build_time_series`, histogram/box_plot/radar/funnel via their `_build_*`); this slice is content (fixture DSL + manifest) + a test refactor. Lowest-risk catalogue work.
- **Per-mode data-shape risk** — some modes need specific config (histogram/box_plot read `value:`; time modes need `bucket()`); the probe-then-pin loop verifies each renders its marker before the entry lands, so no guessed data ships.
- **Drift** — no IR change → no api-surface / golden-master drift. The validate-drift baseline gains 7 region warnings (within the grace).

## Deferred (later B slices)

- List/card modes + config-driven modes (their own coverage batches — each needs items/config).
- Real aggregates over demo_data (replace canned buckets).
- A registry that generates the fixture DSL too (the heavier SSOT step).
