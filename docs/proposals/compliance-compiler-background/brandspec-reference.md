# Brandspec Reference

## Overview

`brandspec.yaml` is a brand identity specification file that extends Dazzle's existing `themespec.yaml` to cover print and compliance document output. While `themespec.yaml` controls web UI styling (OKLCH colour system, CSS custom properties, responsive breakpoints), `brandspec.yaml` provides a unified brand definition that spans both web and print contexts.

**Relationship to themespec.yaml:**

| Aspect | themespec.yaml | brandspec.yaml |
|--------|---------------|----------------|
| **Purpose** | Web UI styling | Unified brand (web + print) |
| **Colours** | OKLCH perceptual system | Hex values (print-safe) |
| **Typography** | CSS font stacks | Font families + print sizes |
| **Layout** | Responsive breakpoints | Page dimensions + margins |
| **Output** | CSS custom properties | PDF rendering parameters |

The compliance renderer (`dazzle.compliance.renderer`) reads `brandspec.yaml` from the project root and falls back to sensible defaults if no file is found. This means projects can generate compliance PDFs immediately without any brand configuration.

## Schema

```yaml
brand:
  # --- Identity ---
  identity:
    name: "Project Name"              # Display name used in headers/footers
    legal_name: "Legal Entity Ltd"     # Used in formal document contexts
    company_number: "12345678"         # Optional: Companies House number
    tagline: "Short description"       # Optional: subtitle on title pages

  # --- Assets ---
  assets:
    logo: "static/images/logo.svg"     # Optional: path to logo file
    favicon: "static/images/favicon.ico"

  # --- Colours ---
  # Hex values for print-safe colour reproduction.
  # Projects using themespec.yaml should derive these from the OKLCH palette.
  colours:
    primary: "#1a365d"       # Main brand colour (headings, table headers)
    secondary: "#2b6cb0"     # Secondary colour (subheadings, accents)
    accent: "#38a169"        # Accent colour (banners, highlights)
    text: "#1a202c"          # Body text colour
    muted: "#718096"         # De-emphasised text (footers, metadata)
    surface: "#f7fafc"       # Background for alternating table rows
    white: "#ffffff"         # Pure white

  # --- Typography ---
  typography:
    headings:
      family: "Source Serif 4, Georgia, serif"
      weight: 600
      line_height: 1.2
    body:
      family: "Inter, system-ui, sans-serif"
      weight: 400
      size: "11pt"
      line_height: 1.7
    mono:
      family: "JetBrains Mono, Fira Code, monospace"

  # --- Print ---
  # Controls PDF page layout. All values use CSS/print units.
  print:
    page_size: A4            # A4, Letter, etc.
    margins:
      top: 25mm
      bottom: 25mm
      left: 30mm             # Wider left for binding
      right: 25mm
    header:
      left: "{document_title}"
      right: "{legal_name}"
      separator: true        # Horizontal rule below header
    footer:
      left: "{legal_name} — Confidential"
      centre: "{document_id} v{version}"
      right: "Page {page} of {pages}"
    styles:
      title_page:
        logo_position: centre
        title_size: 24pt
        subtitle_size: 14pt
        title_font: headings          # References typography.headings
        includes: [document_id, version, date, classification]
      heading_1:
        size: 16pt
        colour: primary               # References colours.primary
        font: headings
        spacing_before: 18pt
      heading_2:
        size: 13pt
        colour: secondary
        font: headings
        spacing_before: 12pt
      heading_3:
        size: 11pt
        colour: text
        font: headings
        weight: 600
      body:
        size: 11pt
        font: body
        colour: text
      table:
        header_bg: primary
        header_text: "#ffffff"
        stripe: "#f7fafc"
        border: muted
      classification_banner:
        text: "CONFIDENTIAL"
        colour: accent
        position: top_centre

  # --- Compliance ---
  # Metadata for document control pages in compliance output.
  compliance:
    document_control:
      author: "Generated by Dazzle Compliance Pipeline"
      reviewer: ""                     # Blank until assigned
      approver: ""                     # Blank until assigned
      classification: "Confidential"
    revision_history: true             # Include revision history table
    distribution_list: true            # Include distribution list
```

## How the Renderer Uses Brandspec

The compliance renderer (`dazzle.compliance.renderer`) uses brandspec values as follows:

1. **`brand.identity`** — populates title page, headers, and footers
2. **`brand.colours`** — injected as CSS custom properties (`--colour-primary`, etc.)
3. **`brand.compliance.document_control`** — fills the document control table on page 2
4. **`brand.typography`** and **`brand.print`** — currently informational; the bundled CSS (`compliance.css`) uses hardcoded values that match the defaults. Future versions will generate CSS dynamically from these sections.

### Fallback Behaviour

If no `brandspec.yaml` is found, the renderer uses a minimal default:

- Organisation name: "Untitled Project"
- Colours: navy/blue/green neutral palette
- Classification: "Confidential"
- Author: "Generated by Dazzle Compliance Pipeline"

This allows any Dazzle project to generate compliance PDFs without configuration.

## Deriving Brandspec from Themespec

Projects that already have a `themespec.yaml` can derive brandspec colours from their OKLCH palette. The mapping is:

| brandspec colour | themespec source |
|-----------------|-----------------|
| `primary` | `brand_hue` at high lightness contrast |
| `secondary` | `brand_hue` at medium lightness |
| `accent` | `brand_hue + accent_hue_offset` |
| `text` | Neutral at low lightness |
| `muted` | Neutral at medium lightness |

Since OKLCH is perceptual, the hex values should be computed once and stored in brandspec rather than converted at runtime.

## Unification Path

Currently Dazzle projects may have both `themespec.yaml` (web) and `brandspec.yaml` (print/compliance). There are two reasonable paths to unification:

### Option A: Brandspec Subsumes Themespec

Add a `web:` section to brandspec that contains the current themespec fields (OKLCH parameters, breakpoints, font stacks). The web server reads `brandspec.yaml` instead of `themespec.yaml`. This is the cleaner long-term approach but requires migration.

### Option B: Themespec Gains a `print:` Section

Add `print:` and `compliance:` sections to `themespec.yaml`. The compliance renderer reads themespec instead of brandspec. This is the lower-friction approach but overloads themespec's original purpose.

**Recommendation:** Option A is preferred. Brand identity is the superset concept; web styling and print styling are both projections of it. A `brandspec.yaml` that contains `web:` (current themespec), `print:`, and `compliance:` sections provides a single source of truth.
